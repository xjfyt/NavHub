# 二、分类系统

> **对应代码**：`backend/src/handlers/groups.rs`、`frontend/src/components/Sidebar.tsx`、`frontend/src/types.ts`（`GroupView`）
> **维护提示**：修改分类 CRUD、推送分类或权限控制逻辑时同步更新本文档。

NavHub 的分类系统（Groups）是导航页的组织骨架。每个分类是一个独立的页面容器，内含图标和小组件。分类支持拖拽排序、推送分发、权限控制等高级功能。

## 1、分类数据模型

### 1.1 核心字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 分类唯一标识 |
| `name` | String | 分类名称 |
| `icon` | String | 分类图标（Lucide 图标名） |
| `owner_id` | UUID? | 所有者用户 ID（`null` = 系统推送） |
| `pushed` | bool | 是否为推送分类 |
| `push_target_type` | Enum | 推送目标类型：`all` / `role` / `user` |
| `push_target_role` | String? | 推送目标角色 |
| `push_target_user_id` | UUID? | 推送目标用户 |
| `push_allow_edit` | bool | 推送分类是否允许用户编辑 |
| `sort_order` | i32 | 全局排序序号 |
| `created_at` | Timestamp | 创建时间 |
| `updated_at` | Timestamp | 更新时间 |

### 1.2 前端视图类型

```typescript
// frontend/src/types.ts
interface GroupView {
  id: string;
  name: string;
  icon: string;
  ownerId: string | null;
  ownerName: string | null;
  pushed: boolean;
  pushTargetType: MessageTargetType;  // "all" | "role" | "user"
  pushTargetRole: string | null;
  pushTargetUserId: string | null;
  pushAllowEdit: boolean;
  sortOrder: number;
  readOnly: boolean;  // 前端计算：推送分类 + 不允许编辑 = 只读
}
```

## 2、分类 CRUD 操作

### 2.1 创建分类

**API 端点**：`POST /api/groups`

**请求体**：

```json
{
  "name": "工作导航",
  "icon": "briefcase"
}
```

**后端逻辑**（[groups.rs#L16-L60](../../backend/src/handlers/groups.rs)）：

1. 验证用户非 Guest 角色
2. `owner_id` 默认为当前用户
3. `sort_order` 自动取该用户分类的最大值 + 1
4. `pushed` 默认为 `false`
5. 记录审计日志

### 2.2 更新分类

**API 端点**：`PATCH /api/groups/:id`

**可修改字段**：`name`、`icon`

**权限校验**：仅分类所有者或管理员可修改。

### 2.3 删除分类

**API 端点**：`DELETE /api/groups/:id`

**级联清理**（DATA-6）：

删除分类后，后端会自动清理 `user_preferences` 中的残留引用：

```sql
UPDATE user_preferences
   SET sidebar_order = array_remove(sidebar_order, $1),
       pushed_group_wallpapers = pushed_group_wallpapers - $2,
       updated_at = now()
 WHERE $1 = ANY(sidebar_order)
    OR jsonb_exists(pushed_group_wallpapers, $2)
```

**设计决策**：偏好清理失败不回滚分类删除（仅告警日志），因为分类删除是更重要的结果。

## 3、推送分类

### 3.1 推送机制

推送分类是管理员创建并分发给用户的分类，典型场景：
- 公司统一导航（推送给所有用户）
- 部门专属导航（推送给特定角色）
- 个人定制导航（推送给特定用户）

```
┌──────────┐    POST /api/admin/groups/:id/push    ┌──────────┐
│  Admin    │ ────────────────────────────────────→ │ Database │
│ (管理台)  │    设置 push_target_* 字段             │ (groups) │
└──────────┘                                        └──────────┘
                                                         │
      ┌──────────────────────────────────────────────────┘
      ▼
┌──────────┐    GET /api/workspace    ┌──────────┐
│  User A   │ ──────────────────────→ │ Handler  │
│ (普通用户) │                         │workspace │
└──────────┘                          └──────────┘
      ↑                                    │
      │        返回 pushed groups           │
      └────────────────────────────────────┘
```

### 3.2 推送目标类型

| 类型 | 说明 | 数据库字段 |
|------|------|-----------|
| `all` | 推送给所有用户 | `push_target_type = 'all'` |
| `role` | 推送给特定角色 | `push_target_role = 'user'` |
| `user` | 推送给特定用户 | `push_target_user_id = '<uuid>'` |

### 3.3 推送分类的编辑权限

- `push_allow_edit = true`：用户可以在推送分类中添加/编辑/删除自己的图标
- `push_allow_edit = false`：分类为只读（`readOnly = true`），用户只能查看

### 3.4 取消推送

**API 端点**：`DELETE /api/admin/groups/:id/push`

将 `pushed` 设为 `false`，清除推送目标字段。

## 4、权限控制

### 4.1 权限判定函数

```rust
// backend/src/handlers/util.rs
pub fn group_writable_by(group: &Group, user: &SessionUser) -> bool {
    // 管理员及以上角色可写任何分类
    if user.role.at_least_admin() {
        return true;
    }
    // Guest 角色永远只读
    if matches!(user.role, Role::Guest) {
        return false;
    }
    // 推送分类：根据 push_allow_edit 判断
    if group.pushed {
        return group.push_allow_edit;
    }
    // 普通分类：仅所有者可写
    group.owner_id == Some(user.id)
}
```

### 4.2 权限矩阵

| 角色 | 自己的分类 | 他人分类 | 推送分类（allow_edit=true） | 推送分类（allow_edit=false） |
|------|-----------|---------|---------------------------|----------------------------|
| Guest | ❌ | ❌ | ❌ | ❌ |
| User | ✅ 读写 | ❌ | ✅ 读写 | 👁️ 只读 |
| Admin | ✅ 读写 | ✅ 读写 | ✅ 读写 | ✅ 读写 |
| SuperAdmin | ✅ 读写 | ✅ 读写 | ✅ 读写 | ✅ 读写 |

### 4.3 只读分类的前端处理

前端通过 `GroupView.readOnly` 字段判断：

- 只读分类不显示"添加图标"按钮
- 只读分类的图标不支持拖拽排序
- 只读分类的图标不支持右键编辑/删除

## 5、分类排序

### 5.1 排序机制

**API 端点**：`POST /api/groups/reorder`

**两种排序模式**：

```
┌─────────────────────────────────────────────────┐
│ 管理员排序                                        │
│  → 更新 groups 表的 sort_order（全局排序）        │
│  → 所有用户看到相同顺序                           │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│ 普通用户排序                                      │
│  → 更新 user_preferences.sidebar_order           │
│  → 仅影响当前用户的侧边栏显示顺序                  │
│  → 不修改 groups 表的全局 sort_order              │
└─────────────────────────────────────────────────┘
```

**后端实现**：

```rust
pub async fn reorder(...) -> AppResult<StatusCode> {
    if user.role.at_least_admin() {
        // 管理员：更新全局 sort_order
        let mut tx = state.pg.begin().await?;
        for (i, gid) in body.order.iter().enumerate() {
            sqlx::query("UPDATE groups SET sort_order = $1 WHERE id = $2")
                .bind(i as i32)
                .bind(gid)
                .execute(&mut *tx)
                .await?;
        }
        tx.commit().await?;
    } else {
        // 普通用户：更新个人偏好
        sqlx::query(
            "INSERT INTO user_preferences (user_id, sidebar_order, updated_at)
             VALUES ($1, $2, now())
             ON CONFLICT (user_id) DO UPDATE SET sidebar_order = $2, updated_at = now()"
        )
        .bind(user.id)
        .bind(&body.order)
        .execute(&state.pg)
        .await?;
    }
    Ok(StatusCode::NO_CONTENT)
}
```

### 5.2 侧边栏排序持久化

用户自定义排序存储在 `user_preferences.sidebar_order`（UUID 数组），前端加载工作区时：
1. 读取用户的 `sidebar_order`
2. 按该顺序排列分类
3. 未在 `sidebar_order` 中的分类追加到末尾

## 6、分类内元素排序

### 6.1 混合排序

分类内的图标和小组件共享同一个排序空间。

**API 端点**：`POST /api/groups/:id/reorder-items`

**请求体**：

```typescript
interface GroupItemsReorderRequest {
  order: {
    id: string;          // 图标或小组件 ID
    type: "icon" | "widget";
    x?: number | null;   // 可选的网格坐标
    y?: number | null;
  }[];
}
```

**安全措施**：
- 使用 PostgreSQL Advisory Lock（`pg_advisory_xact_lock`）防止并发 reorder 死锁
- 校验 payload 大小不超过数据库中的实际元素数量
- 检测并拒绝重复条目

### 6.2 并发安全

```rust
// 使用事务 + Advisory Lock 确保并发安全
sqlx::query("SELECT pg_advisory_xact_lock(hashtext($1::text))")
    .bind(id.to_string())
    .execute(&mut *tx)
    .await?;
```

## 7、前后端交互总结

| 操作 | 触发方式 | API 端点 | 特殊逻辑 |
|------|----------|----------|----------|
| 创建分类 | 侧边栏"+ 添加" | `POST /api/groups` | 自动排序末尾 |
| 编辑分类 | 右键菜单 → 编辑 | `PATCH /api/groups/:id` | 仅 name/icon |
| 删除分类 | 右键菜单 → 删除 | `DELETE /api/groups/:id` | 级联清理偏好 |
| 排序分类 | 侧边栏拖拽 | `POST /api/groups/reorder` | 区分管理员/用户 |
| 推送分类 | 管理台 → 推送 | `POST /api/admin/groups/:id/push` | 设置目标+权限 |
| 取消推送 | 管理台 → 取消 | `DELETE /api/admin/groups/:id/push` | 清除推送标记 |
