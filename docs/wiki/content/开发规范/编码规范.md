# Rust和TypeScript编码规范

> **对应代码**：全项目
> **维护提示**：修改编码规范时同步更新本文档。

本文档定义NavHub项目的编码规范，涵盖Rust后端和TypeScript前端。

## 1. Rust 编码规范

### 1.1 命名规范

| 元素 | 风格 | 示例 |
|------|------|------|
| 模块名 | snake_case | `user_prefs` |
| 类型名 | PascalCase | `UserConfig` |
| 函数/变量 | snake_case | `get_user_prefs` |
| 常量 | SCREAMING_SNAKE_CASE | `MAX_RETRIES` |
| 生命周期 | 短名 | `'a`、`'de` |

### 1.2 序列化规范

所有面向前端的serde结构体使用：

```rust
#[serde(rename_all = "camelCase")]
```

与前端TypeScript类型对齐。

### 1.3 错误处理

- 使用 `anyhow::Result` 处理应用错误
- 使用 `thiserror` 定义模块级错误类型
- **禁止**在生产代码中使用 `unwrap()`（可用 `expect("reason")` 用于初始化）

```rust
// 正确
let value = config.get("key").ok_or_else(|| anyhow!("missing key"))?;

// 错误（生产代码中）
let value = config.get("key").unwrap();
```

### 1.4 异步编程

- `PgPool`（sqlx）是 Send + Sync：可安全在异步上下文中共享
- `redis::aio::Connection` 非 Sync：需在专用任务中使用
- 避免在异步函数中执行阻塞操作，使用 `tokio::task::spawn_blocking`

### 1.5 注释规范

- 模块级：`//!` 文档注释
- 公共API：`///` 文档注释
- 关键逻辑：行内注释说明"为什么"而非"是什么"

### 1.6 代码格式化

项目使用 `rustfmt` 格式化代码，配置见 `backend/rustfmt.toml`：

```bash
cargo fmt           # 格式化
cargo fmt --check   # 检查（CI用）
cargo clippy        # 静态分析
```

## 2. TypeScript 编码规范

### 2.1 命名规范

| 元素 | 风格 | 示例 |
|------|------|------|
| 组件 | PascalCase | `WorkspaceScreen` |
| 函数/变量 | camelCase | `fetchGroups` |
| 常量 | SCREAMING_SNAKE_CASE | `API_BASE_URL` |
| 类型/接口 | PascalCase | `GroupConfig` |

### 2.2 组件规范

- 使用函数组件 + Hooks
- Props接口以 `Props` 后缀命名
- 导出方式：`export default function ComponentName()`

```typescript
interface CardProps {
  title: string;
  count: number;
}

export default function Card({ title, count }: CardProps) {
  return <div>{title}: {count}</div>;
}
```

### 2.3 状态管理

- 使用React内置状态管理（useState、useReducer）
- 避免过度嵌套状态
- 使用Context共享跨组件数据

## 3. 安全规范

### 3.1 输入验证

- 验证所有用户输入
- 使用参数化查询防止SQL注入（sqlx自动处理）
- 前端输出时转义防止XSS

### 3.2 认证授权

- 使用安全的密码哈希（bcrypt/argon2）
- 实施基于角色的权限控制
- 记录安全相关操作日志
