# 文档编写规范

> **对应代码**：`docs/`、`docs/wiki/`
> **维护提示**：文档规范变更时同步更新本文档。

本文档定义NavHub项目的文档编写规范，确保文档质量和一致性。

## 1. 文档结构

### 1.1 文档分类

| 目录 | 用途 |
|------|------|
| `docs/` | 官方技术文档（版本化） |
| `docs/wiki/` | Wiki知识库（快速更新） |
| 代码内注释 | 模块级和API级文档 |

### 1.2 Wiki目录结构

```
docs/wiki/content/
  项目概述/      # 项目介绍和架构概览
  快速开始/      # 入门指南
  后端架构/      # 后端技术架构
  前端架构/      # 前端技术架构
  功能模块/      # 各功能模块详解
  运维指南/      # 部署和运维
  开发规范/      # 编码和工作流规范
```

## 2. 文件格式规范

### 2.1 文件头元数据

每个wiki文档必须包含YAML front matter：

```yaml
---
title: 文档标题
description: 一句话描述
keywords: 关键词1, 关键词2
author: NavHub Team
last_updated: YYYY-MM-DD
category: 目录分类
nav_order: 排序号
---
```

### 2.2 文件尾导航

每个文档末尾必须包含导航链接：

```markdown
---

- 上一篇：[前一篇标题](./前一篇.md)
- 下一篇：[后一篇标题](./后一篇.md)
- 返回目录：[目录名](./README.md)
```

### 2.3 内容格式

- 使用中文编写，专业术语可保留英文
- 标题层级：h1（文档标题）> h2（主要章节）> h3（子章节）
- 代码块必须标注语言类型
- 表格用于结构化数据对比
- 列表用于步骤和要点

## 3. 文档同步规则

### 3.1 代码变更时

- 修改代码后**必须**更新对应文档
- 新增模块时创建对应文档
- 删除功能时清理相关文档

### 3.2 文档头声明

每份文档头部必须声明：

> **对应代码**：相关的代码文件路径
> **维护提示**：何时需要更新本文档

## 4. 编写最佳实践

### 4.1 受众导向

- **快速开始**：面向新用户，简洁明了
- **架构文档**：面向开发者，技术细节充分
- **运维指南**：面向运维人员，包含实际命令
- **API文档**：面向调用方，参数和示例完整

### 4.2 示例代码

- 所有代码示例必须可运行
- 包含必要的上下文和注释
- 使用占位符标注需要替换的值

```bash
# 正确：标注占位符
PG_USER=your_username PG_PASSWORD=your_password ./scripts/backup.sh

# 错误：硬编码真实凭据
PG_USER=admin PG_PASSWORD=123456 ./scripts/backup.sh
```

### 4.3 更新频率

- 核心架构文档：随架构变更更新
- API文档：随接口变更更新
- 运维指南：随部署配置变更更新
- FAQ/故障排查：遇到新问题时补充

## 5. 文档审查

### 5.1 审查清单

- [ ] YAML front matter完整且正确
- [ ] 文件尾导航链接正确
- [ ] 代码示例可运行
- [ ] 中英文混排有空格
- [ ] 无敏感信息泄露（密码、密钥等）
- [ ] 与实际代码/配置一致
