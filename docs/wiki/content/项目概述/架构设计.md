# 三、架构设计

> **对应代码**：`backend/src/`、`frontend/src/`、`Dockerfile`、`docker-compose.yaml`
> **维护提示**：架构变更（新增模块、修改数据流、调整部署形态）时同步更新本文档。

## 1、整体架构概述

NavHub 采用**单进程一体化部署**架构，Rust 后端同时承担 API 服务和前端静态资源托管职责。

```
+-----------------------------------------------------+
|                    Browser / Client                  |
|                  React 18 + TypeScript               |
+-------------+-----------------------+---------------+
              | /auth/*               | /api/*
              v                       v
+-----------------------------------------------------+
|               Rust 后端 (Axum, 单进程 :8088)         |
|                                                     |
|  +----------+  +----------+  +------------------+  |
|  | Auth 模块 |  | API 模块  |  | 静态资源托管模块  |  |
|  | /auth/*  |  | /api/*   |  | / (前端 dist)    |  |
|  +----+-----+  +----+-----+  +------------------+  |
|       |              |                               |
|  +----+--------------+-------------------------+    |
|  |            中间件层 (Middleware)              |    |
|  |  认证校验 | 权限控制 | 限流 | 请求ID | 日志   |    |
|  +----+--------------+-------------------------+    |
+-------+--------------+------------------------------+
        |              |
   +----v----+   +-----v-----+   +--------------+
   | Redis 7 |   |PostgreSQL |   |  S3 / MinIO  |
   | session |   |    15     |   |  文件存储     |
   | cache   |   |  主数据    |   |  图标/壁纸    |
   +---------+   +-----------+   +--------------+
```

### 1.1 四大路由职责

| 路由前缀 | 职责 | 说明 |
|----------|------|------|
| `/api/*` | REST API | 所有业务接口（CRUD、管理、搜索等） |
| `/auth/*` | 认证回调 | OIDC 回调处理、密码登录、登出 |
| `/uploads/*` | 用户上传 | 托管用户上传的图标图片 |
| `/*`（其他） | 前端 SPA | 托管 `frontend/dist/` 的静态文件 |

### 1.2 单进程优势

- **部署简单**：一个 Docker 容器包含所有服务
- **一致性保证**：前后端版本天然同步
- **资源节省**：无需独立的 Nginx 或 Node.js 服务

## 2、后端模块架构

### 2.1 目录结构

```
backend/src/
+-- main.rs              # 入口：初始化连接池、路由、启动服务
+-- config.rs            # 配置加载（config.toml + 环境变量覆盖）
+-- db.rs                # PostgreSQL 连接池
+-- cache.rs             # Redis 连接池
+-- error.rs             # 统一错误类型（AppError -> HTTP 响应）
+-- routes.rs            # 路由注册
+-- state.rs             # 共享应用状态
+-- storage.rs           # S3 存储客户端
+-- tasks.rs             # 后台定时任务（壁纸抓取、清理等）
+-- request_id.rs        # 请求 ID 中间件
|
+-- auth/                # 认证模块
|   +-- mod.rs           # 模块入口
|   +-- casdoor.rs       # OIDC 客户端（Casdoor 动态配置）
|   +-- oidc.rs          # 通用 OIDC 处理
|   +-- password.rs      # Argon2 密码哈希/校验
|   +-- session.rs       # Session 管理（Redis 存储）
|   +-- middleware.rs     # 认证中间件（提取当前用户）
|   +-- rate_limit.rs    # 登录限流
|   +-- sso_cache.rs     # SSO 配置缓存
|
+-- handlers/            # 请求处理器
|   +-- auth.rs          # 登录/登出/回调处理
|   +-- me.rs            # 当前用户信息
|   +-- workspace.rs     # 工作区管理
|   +-- groups.rs        # 分组 CRUD
|   +-- icons.rs         # 图标 CRUD
|   +-- prefs.rs         # 用户偏好设置
|   +-- upload.rs        # 文件上传
|   +-- favicon.rs       # Favicon 代理
|   +-- wallpapers.rs    # 壁纸管理
|   +-- admin/           # 管理后台接口
|
+-- models/              # 数据模型
|   +-- user.rs          # 用户模型
|   +-- group.rs         # 分组模型
|   +-- icon.rs          # 图标模型
|   +-- widget.rs        # 小组件模型
|   +-- prefs.rs         # 偏好设置模型
|   +-- wallpaper.rs     # 壁纸模型
|   +-- audit.rs         # 审计日志模型
|
+-- scraper/             # 壁纸/图标抓取器
    +-- bing.rs          # Bing 壁纸
    +-- nasa.rs          # NASA 每日图片
    +-- unsplash.rs      # Unsplash
    +-- wallhaven.rs     # Wallhaven
    +-- pexels.rs        # Pexels
    +-- pixabay.rs       # Pixabay
    +-- wikimedia.rs     # Wikimedia Commons
    +-- iconify.rs       # Iconify 图标库
```

### 2.2 数据流

```
(1) 用户登录
Browser --GET /auth/login--> Axum
         --302 redirect----> Casdoor (OIDC Provider)
         <--callback?code-- Casdoor
Axum --> 校验 code --> 获取 userinfo --> 写入 Redis session
         --Set-Cookie: nh_sid--> Browser

(2) API 请求
Browser --GET /api/groups--> Axum
Middleware: 验证 nh_sid cookie --> Redis 查询 session
Handler: 读取 PostgreSQL --> 返回 JSON
```

### 3.2 开发模式代理

Vite 开发服务器配置了 API 代理，将 `/api` 和 `/auth` 请求转发到后端：

```typescript
// frontend/vite.config.ts
export default defineConfig({
  server: {
    proxy: {
      '/api': 'http://127.0.0.1:8088',
      '/auth': 'http://127.0.0.1:8088',
      '/uploads': 'http://127.0.0.1:8088',
    },
  },
});
```

## 4、数据库设计

### 4.1 核心表结构

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| `users` | 用户表 | id, username, email, role, password_hash |
| `groups` | 分组表 | id, user_id, name, sort_order |
| `icons` | 图标表 | id, group_id, title, url, icon_style, grid_x, grid_y |
| `icon_assets` | 图标资源表 | id, icon_library, name, svg_data |
| `widgets` | 小组件表 | id, user_id, type, config, position |
| `user_prefs` | 用户偏好 | id, user_id, wallpaper, theme, locale |
| `messages` | 系统消息 | id, title, content, expires_at |
| `audit_log` | 审计日志 | id, user_id, action, target, created_at |
| `remote_wallpapers` | 远程壁纸 | id, source_id, external_id, url, thumbnail_url |
| `wallpaper_sources` | 壁纸数据源 | id, name, type, base_url |

### 4.2 迁移管理

数据库迁移文件位于 `backend/migrations/`，使用数字编号排序：

```
001_init.sql        # 初始化表结构
002_seed.sql        # 种子数据
003_messages.sql    # 消息表
...
032_expires_partial_indexes.sql  # 最新迁移
```

**迁移规则：**

- 启动时自动执行未运行的迁移
- 新增迁移文件必须递增编号
- 迁移必须是幂等的（可重复执行）

## 5、权限模型

### 5.1 角色定义

| 角色 | 说明 | 登录方式 |
|------|------|----------|
| `superadmin` | 超级管理员，拥有所有权限 | OIDC + 账号密码 |
| `user` | 普通用户，管理自己的数据 | 仅 OIDC |

### 5.2 数据隔离

- 每个用户拥有独立的**工作区**（workspace）
- 分组、图标、小组件、偏好设置按 `user_id` 隔离
- 超级管理员可查看/管理所有用户数据（通过后台接口）

## 6、部署架构

### 6.1 Docker 容器

NavHub 使用**多阶段构建**的单一 Docker 镜像：

- **阶段 1**：使用 `node:20-alpine` 构建前端产物
- **阶段 2**：使用 `rust:1.95` 构建后端二进制
- **阶段 3**：使用 `debian:bookworm-slim` 作为最终运行镜像

### 6.2 反向代理

生产环境推荐使用 Nginx 作为反向代理：

```nginx
server {
    listen 443 ssl;
    server_name navhub.example.com;

    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    location / {
        proxy_pass http://127.0.0.1:8088;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```
