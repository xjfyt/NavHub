# 性能调优建议

> **对应代码**：`backend/src/cache.rs`、`backend/src/config.rs`、`backend/src/db.rs`
> **维护提示**：性能相关配置变更时同步更新本文档。

本文档提供NavHub项目的性能调优建议，涵盖后端、数据库、前端和基础设施层面。

## 1. 后端性能优化

### 1.1 缓存策略

NavHub使用Redis作为缓存层，主要缓存：

- **会话数据**：用户登录状态和权限信息
- **热点数据**：频繁访问的分组、图标列表
- **限流计数**：API请求频率限制

缓存失效策略：
- 写操作时主动清除相关缓存
- 设置合理的TTL（Time To Live）
- 使用缓存预热机制减少冷启动影响

### 1.2 数据库连接池

```toml
# config.toml
[database]
max_connections = 20      # 连接池大小
min_connections = 5       # 最小空闲连接
connect_timeout = 30      # 连接超时（秒）
idle_timeout = 600        # 空闲超时（秒）
```

建议：连接池大小 = CPU核心数 * 2 + 磁盘数

## 2. 数据库性能优化

### 2.1 索引策略

NavHub的迁移文件已包含关键索引（见 `012_indexes.sql`、`025_cleanup_and_indexes.sql`）。常见查询模式：

- 分组图标查询：`(group_id, sort_order)` 复合索引
- 用户偏好查询：`(user_id)` 索引
- 消息时间范围查询：`(created_at)` 部分索引（仅未过期）

### 2.2 查询优化建议

- 避免 SELECT *，只查询需要的字段
- 大量数据分页使用游标分页（cursor pagination）而非OFFSET
- 批量操作使用事务包裹减少往返
- 定期执行 VACUUM ANALYZE 更新统计信息

### 2.3 PostgreSQL 调优参数

```toml
# 典型 2核4G 服务器配置
shared_buffers = 1GB
effective_cache_size = 3GB
work_mem = 32MB
maintenance_work_mem = 256MB
max_connections = 100
```

## 3. 前端性能优化

### 3.1 静态资源缓存

NavHub已内置优化：
- `assets/` 下的CSS/JS资源：`max-age=31536000`（1年）+ immutable
- `index.html`：`no-cache` 保证入口随时更新

### 3.2 构建优化

- Vite自动代码分割和Tree Shaking
- 图片懒加载和响应式处理
- CSS/JS压缩和Gzip/Brotli预压缩

## 4. 基础设施优化

### 4.1 Docker资源限制

根据 `docker-compose.yaml`，建议的资源分配：

| 服务 | 内存 | CPU | 说明 |
|------|------|-----|------|
| navhub | 512MB | 1.0 | 应用服务 |
| postgres | 512MB | 1.0 | 数据库 |
| redis | 256MB | 0.5 | 缓存 |

### 4.2 反向代理优化

Nginx性能相关配置：

```nginx
# 启用HTTP/2
listen 443 ssl http2;

# 启用Gzip压缩
gzip on;
gzip_types text/plain application/json application/javascript text/css;

# 静态资源直接由Nginx处理
location /assets/ {
    proxy_pass http://backend;
    expires 1y;
    add_header Cache-Control "public, immutable";
}
```

### 4.3 监控指标

关注以下关键指标：
- API响应时间P50/P95/P99
- 数据库连接池使用率
- Redis内存使用率和命中率
- 容器CPU和内存使用率
