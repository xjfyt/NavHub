# 数据备份与恢复策略

> **对应代码**：`scripts/backup.sh`、`scripts/restore.sh`
> **维护提示**：备份恢复流程变更时同步更新本文档。

本手册说明NavHub的数据备份、恢复与演练流程。

## 1. 需要持久化的数据

1. **PostgreSQL数据库** — 用户、分组、图标、偏好、消息等全部业务数据
2. **对象/上传存储** — 头像、上传的图标、壁纸等二进制资产
   - S3/对象存储：依赖桶的版本控制+跨区域复制，无需脚本备份
   - 本地存储：用 `scripts/backup.sh` 的 `NAVHUB_UPLOADS_DIR` 一并打包

> Redis仅作缓存/限流，**无需备份**（丢失后自动重建）。

## 2. 备份

脚本：`scripts/backup.sh`（使用 `pg_dump -Fc` 自定义格式）。

```bash
# 直连数据库（宿主已装 postgresql-client）
PG_HOST=localhost PG_USER=navhub PG_PASSWORD=xxx PG_DATABASE=navhub \
  BACKUP_DIR=/srv/navhub/backups RETENTION=14 \
  ./scripts/backup.sh

# docker-compose 部署
USE_DOCKER_COMPOSE=1 PG_PASSWORD=xxx ./scripts/backup.sh

# 同时打包本地上传目录
NAVHUB_UPLOADS_DIR=/srv/navhub/uploads PG_PASSWORD=xxx ./scripts/backup.sh
```

脚本自动读取仓库根目录的 `.env`（若存在），输出形如 `navhub-db-YYYYmmdd-HHMMSS.dump`，并按 `RETENTION` 清理过旧备份。

### 定时备份（cron示例）

```cron
# 每天 03:30 备份，保留最近14份
30 3 * * * cd /srv/navhub && USE_DOCKER_COMPOSE=1 RETENTION=14 \
  PG_PASSWORD=xxx /srv/navhub/scripts/backup.sh >> /var/log/navhub-backup.log 2>&1
```

> 把备份**异地同步**（rsync/对象存储）才算真正安全——本机磁盘损坏会同时带走备份。

## 3. 恢复

脚本：`scripts/restore.sh`（使用 `pg_restore --clean --if-exists -1`，单事务，失败整体回滚）。

> **破坏性操作**，默认需交互确认。

```bash
USE_DOCKER_COMPOSE=1 PG_PASSWORD=xxx \
  ./scripts/restore.sh /srv/navhub/backups/navhub-db-20260529-033000.dump

# 自动化场景可用 FORCE=1 跳过确认

# 本地上传目录恢复（如有打包）
tar -xzf navhub-uploads-20260529-033000.tar.gz -C /srv/navhub/
```

恢复后：**重启navhub**，然后验证登录、分组/图标数量、近期数据是否齐全。

## 4. 恢复演练清单（建议每季度一次）

> 备份从未被恢复过 = 没有备份。在**独立的演练环境**执行，切勿直接恢复到生产。

- [ ] 取一份最近的生产备份，验证文件未损坏（`pg_restore -l <dump>` 能列出目录）
- [ ] 在隔离环境执行 `restore.sh`
- [ ] 启动应用指向恢复后的库，确认迁移完整性校验通过
- [ ] 用管理员账号登录，抽查用户列表、分组与图标、系统消息
- [ ] 若有本地上传：抽查头像/图标资源可正常加载
- [ ] 记录实际恢复耗时（RTO）与可接受的数据丢失窗口（RPO）
- [ ] 演练结束销毁演练环境与其中的数据副本

## 5. 多副本与迁移

- **单实例**：无需特殊处理
- **多副本**：首次上线含新迁移的版本时，先把navhub缩到1副本完成迁移再扩容
- 迁移文件checksum不一致时启动会告警但继续；设 `NAVHUB_MIGRATION_STRICT=1` 可强制阻断
