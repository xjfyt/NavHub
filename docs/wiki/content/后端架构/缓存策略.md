# 四、缓存策略

> **对应代码**：`backend/src/cache.rs`、`backend/src/auth/session.rs`、`backend/src/auth/rate_limit.rs`、`backend/src/state.rs`
> **维护提示**：修改缓存策略、Redis 使用方式时同步更新本文档。

## 1、Redis 连接池

NavHub 使用 `deadpool-redis` 管理 Redis 连接池，配置如下：

```rust
const REDIS_POOL_MAX_SIZE: usize = 32;        // 最大连接数
const REDIS_WAIT_TIMEOUT: Duration = 3s;      // 从池中取连接超时
const REDIS_CREATE_TIMEOUT: Duration = 3s;    // 创建新连接超时
```

Redis 连接 URL 由配置生成，支持密码认证和数据库选择。Redis 不可用时请求快速失败（3 秒超时），而非无限等待。

## 2、缓存用途

### （1）会话存储

会话数据是 Redis 最核心的用途，详见 [认证系统.md](./认证系统.md) §3。

- **键格式**：`session:{sid}`（48 字符随机 ID）
- **值**：JSON 序列化的 SessionData
- **TTL**：7 天（可配置）
- **用户会话索引**：`user_sessions:{uid}` SET 结构，用于强制下线

### （2）登录限流

使用 Redis INCR + EXPIRE 实现滑动窗口限流：

- **IP 限流**：`rl:auth_pwd:{ip}`（60s/10次）
- **账户限流**：`rl:auth_acct:{identity}`（300s/10次）
- **实现**：原子 Pipeline（INCR + EXPIRE），一次网络往返

### （3）活跃状态追踪

通过 Redis SET NX EX 实现每分钟至多一次的 last_seen_at 更新：

```rust
// SET user:seen:{uid} 1 NX EX 60
// 键不存在时写入（首次访问或 TTL 过期后），成功后才 UPDATE users
```

### （4）Favicon 单飞（Single-Flight）

使用进程内 HashMap<String, Arc<Notify>> 实现 favicon 抓取的 single-flight，
避免缓存击穿时大量并发请求同时抓取同一域名的 favicon。

## 3、Redis 键一览

| 键模式 | 类型 | TTL | 用途 |
|--------|------|-----|------|
| session:{sid} | String | 7 天 | 会话数据 |
| user_sessions:{uid} | Set | 7 天 | 用户会话索引 |
| rl:auth_pwd:{ip} | String | 60 秒 | IP 限流计数 |
| rl:auth_acct:{identity} | String | 300 秒 | 账户限流计数 |
| user:seen:{uid} | String | 60 秒 | 活跃状态标记 |

## 4、容错策略

- **连接池超时**：3 秒获取/创建超时，Redis 不可用时快速失败
- **会话降级**：Redis 不可用时，会话读取返回 None，用户被要求重新登录
- **限流失效**：Redis 不可用时，限流检查被跳过（fail-open），避免阻塞正常登录
- **last_seen 更新**：Redis 或 DB 失败时静默忽略，不影响主流程
