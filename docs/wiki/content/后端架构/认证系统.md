# 一、认证系统

> **对应代码**：`backend/src/auth/mod.rs`、`backend/src/auth/oidc.rs`、`backend/src/auth/password.rs`、`backend/src/auth/session.rs`、`backend/src/auth/middleware.rs`、`backend/src/auth/rate_limit.rs`、`backend/src/auth/sso_cache.rs`、`backend/src/auth/casdoor.rs`
> **维护提示**：修改认证逻辑、新增认证方式、变更会话策略时同步更新本文档。

## 1、模块结构

```
backend/src/auth/
├── mod.rs           # 模块入口、权限检查函数
├── casdoor.rs       # Casdoor OIDC 客户端集成
├── oidc.rs          # OIDC 协议核心实现（PKCE、JWT 验证、JWKS 缓存）
├── password.rs      # Argon2 密码哈希与验证
├── session.rs       # Redis 会话管理
├── middleware.rs     # require_login / optional_login 中间件
├── rate_limit.rs    # 基于 Redis 的登录限流
└── sso_cache.rs     # 进程内 SSO 配置缓存（带 TTL）
```

## 2、认证方式

NavHub 支持两种认证方式：**账号密码认证**（仅限超级管理员）和 **OIDC 单点登录**（所有用户）。

### （1）账号密码认证

密码认证仅面向超级管理员，使用 Argon2 算法进行密码哈希。相关代码见 `backend/src/auth/password.rs`：

```rust
pub fn hash_password(password: &str) -> anyhow::Result<String>
pub fn verify_password(password: &str, hash: &str) -> bool
```

**首次启动**：系统自动引导创建超级管理员账户（见 `bootstrap_superadmin`）。
若配置 `force_change_password = true`，首次登录后强制修改密码。

### （2）OIDC 单点登录

OIDC 认证流程采用 PKCE（RFC 7636）增强安全性，完整流程如下：

```
┌──────────┐     ┌──────────┐     ┌──────────────┐     ┌──────────────┐
│  浏览器   │ ──→ │  /auth/   │ ──→ │  OIDC 提供商  │ ──→ │  /auth/      │
│          │     │  login    │     │  (Casdoor等)  │     │  callback    │
└──────────┘     └──────────┘     └──────────────┘     └──────────────┘
```

**OAuth 流程绑定（AUTH-7）**：每次登录生成 OauthFlow 结构体，包含三个密码学随机值：

```rust
pub struct OauthFlow {
    pub state: String,         // CSRF 防御
    pub nonce: String,         // ID Token 重放防御
    pub code_verifier: String, // PKCE 授权码拦截防御
}
```

该结构序列化后存入 `nh_oauth` HttpOnly Cookie，有效期 600 秒。

**ID Token 验证（AUTH-1）**：

| 验证步骤 | 说明 |
|---------|------|
| 签名验证 | 使用 JWKS 中对应 kid 的 RSA 公钥验证 RS256 签名 |
| iss 验证 | 与配置的 OIDC issuer 匹配 |
| aud 验证 | 与配置的 client_id 匹配 |
| exp 验证 | 含 60 秒时钟偏移容差（leeway） |
| nonce 验证 | 必须等于浏览器绑定的 nonce 值 |

JWKS 公钥集缓存在进程内，TTL 为 300 秒，当 kid 未命中或缓存过期时自动重新获取。

## 3、会话管理

### （1）会话存储

会话数据存储在 Redis 中，采用 `session:{sid}` 作为键格式：

```rust
pub struct SessionData {
    pub user_id: Uuid,
    pub role: String,
    pub username: String,
    pub email: String,
    pub must_change_password: bool,
}
```

会话 ID 生成：48 字符的字母数字随机串，使用 `rand::thread_rng()` 生成。

### （2）会话生命周期

| 操作 | Redis 命令 | 说明 |
|------|-----------|------|
| 创建 | SET session:{sid} <json> EX <ttl> | 存储会话数据 |
| 创建索引 | SADD user_sessions:{uid} {sid} | 按用户查所有会话 |
| 读取 | GET session:{sid} | 反序列化 SessionData |
| 销毁单个 | DEL session:{sid} | 登出当前会话 |
| 销毁全部 | SMEMBERS → 逐个 DEL | 强制下线所有设备 |

**TTL**：由 `app.session_ttl_days` 配置，默认 7 天。

### （3）Cookie 设置

| 属性 | 值 | 说明 |
|------|-----|------|
| 名称 | nh_sid | 固定值 |
| HttpOnly | 是 | 防止 JS 读取 |
| SameSite | Lax | 允许 SSO 回调跨站导航携带 |
| Secure | 条件性 | 仅当 public_url 为 https 时启用 |
| Path | / | 全路径有效 |

> **SameSite=Lax 选择原因**：OIDC 回调从外部 IdP 重定向回来属于顶级导航，Strict 会导致 Cookie 丢失。

## 4、认证中间件

### （1）require_login

强制认证中间件，应用于 `/api` 下所有需要登录的路由。从 Cookie 提取 `nh_sid`，查询 Redis 会话，失败则返回 401。

特殊处理：当 `must_change_password = true` 时，仅允许访问 `/auth/password/change` 端点。

### （2）optional_login

可选认证中间件，用于访客可见的路由（workspace、wallpapers、favicon）。有 Cookie 且有效时注入 Some(SessionUser)，否则注入 None。

### （3）last_seen 更新

认证成功后通过 Redis `SET ... NX EX 60` 实现**每分钟至多一次**的 `last_seen_at` 更新，避免高频写库。

## 5、权限控制

### （1）角色系统

```rust
pub enum Role {
    Superadmin, // 超级管理员，拥有所有权限
    Admin,      // 管理员，拥有后台管理权限
    User,       // 普通用户，只能编辑自有内容
    Guest,      // 访客，只读权限
}
```

### （2）路由权限映射

| 路由分组 | 认证要求 | 权限 |
|---------|----------|------|
| /api/workspace、/api/wallpapers、/api/favicon | optional_login | 访客可访问 |
| /api/me/*、/api/groups/*、/api/icons/* | require_login | 登录用户 |
| /api/admin/* | require_login | 需 at_least_admin() |
| /auth/login、/auth/callback、/auth/password | 无 | 公开端点 |

## 6、登录限流

### （1）双维度限流（AUTH-4）

| 维度 | 键格式 | 窗口 | 最大次数 |
|------|--------|------|----------|
| IP 地址 | rl:auth_pwd:{ip} | 60 秒 | 10 次 |
| 账户标识 | rl:auth_acct:{identity} | 300 秒 | 10 次 |

两个维度独立计数，攻击者轮换 IP 也无法绕过账户维度限流。

### （2）可信代理（AUTH-4）

当直连 peer 在 `trusted_proxies` 配置列表中时，才信任 `X-Forwarded-For` 头。支持单 IP 和 CIDR 格式。

## 7、SSO 配置缓存（OPS-11）

SSO 配置缓存在进程内 `RwLock<CachedSso>` 中，TTL 为 30 秒。

- **快路径**：读锁判断是否陈旧 → 未陈旧直接返回克隆
- **慢路径**：陈旧时在锁外重载 DB → 双重检查后写入缓存
- **失败回退**：重载失败时返回旧缓存（fail-open），避免 DB 抖动阻断认证路径

## 8、HTTP 客户端策略

| 客户端 | TLS 验证 | 重定向 | 用途 |
|--------|----------|--------|------|
| reqwest_client | ✅ 严格 | ✅ 跟随 | 天气、热搜等公网 API |
| lenient_client | 可配置 | ❌ 禁止 | favicon 代理、自签名 CA |
| oidc_client | ✅ 严格（不可降级） | ❌ 禁止 | OIDC 令牌交换、JWKS 获取 |

> **安全设计**：`oidc_client` 独立于 `tls_accept_invalid_certs` 配置，即使开启自签名支持，OIDC 安全路径也不会降级。
