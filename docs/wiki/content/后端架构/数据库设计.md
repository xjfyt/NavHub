# 二、数据库设计

> **对应代码**：`backend/src/db.rs`、`backend/src/models/`、`backend/migrations/`
> **维护提示**：修改数据库 schema、迁移逻辑、数据模型时同步更新本文档。

## 1、技术选型

NavHub 使用 PostgreSQL 作为主数据库，通过 SQLx 进行编译时校验的数据库访问。

| 组件 | 技术 | 说明 |
|------|------|------|
| 数据库 | PostgreSQL 14+ | 支持 JSONB、pgcrypto 扩展 |
| ORM/驱动 | SQLx | 编译时 SQL 校验、异步 |
| 连接池 | SQLx PgPool | 可配置连接数、超时 |
| 迁移 | SQLx Migrator | 编号顺序执行、宽松模式 |

## 2、连接管理

### （1）连接池配置

```rust
PgPoolOptions::new()
    .max_connections(cfg.max_connections)  // 最大连接数，默认 10
    .min_connections(1)                    // 最小连接数
    .acquire_timeout(Duration::from_secs(10))  // 获取连接超时
    .idle_timeout(Some(Duration::from_secs(300)))  // 空闲超时 5min
    .max_lifetime(Some(Duration::from_secs(1800))) // 最大生命周期 30min
    .test_before_acquire(true)  // 获取前测试连接可用性
```

### （2）自动创建数据库

如果目标数据库不存在，系统会自动连接到默认的 postgres 数据库执行 CREATE DATABASE。

### （3）安全策略

关闭 SQLx 语句日志和慢查询日志，避免在 Debug 级别泄露密码 hash、邮箱等 PII。

## 3、数据库迁移

### （1）迁移文件

迁移文件位于 `backend/migrations/` 目录，从 001 到 032 按编号顺序执行。

### （2）迁移策略

系统采用宽松的迁移策略，与 SQLx 默认的严格行为不同：

- **跳过已应用**：版本号已记录在 `_sqlx_migrations` 中的迁移直接跳过
- **重复版本检测**：磁盘上同一版本号出现多次时，生产环境 FAIL、开发环境 warn
- **Checksum 校验**：已记录 checksum 与磁盘文件不一致时默认 warn，设 `NAVHUB_MIGRATION_STRICT=1` 可升级为 FAIL
- **数据库记录但磁盘缺失**：不视为错误（可能是历史迁移已被清理）

### （3）完整性检查

```rust
fn check_migration_integrity(
    on_disk: &[(i64, &[u8])],      // 磁盘上的版本和 checksum
    recorded: &[(i64, Vec<u8>)],     // 数据库已记录的版本和 checksum
) -> Result<(), Vec<String>>
```

## 4、数据模型

### （1）核心表结构

| 表名 | 说明 | 关键字段 |
|------|------|----------|
| users | 用户表 | id, username, email, role, password_hash |
| groups | 分类/分组 | id, name, owner_id, pushed, sort_order |
| icons | 图标/书签 | id, group_id, name, url, image_url, is_folder |
| folder_items | 文件夹内子项 | id, folder_icon_id, name, url |
| widgets | 小组件 | id, group_id, widget_type, config(JSONB) |
| user_preferences | 用户偏好 | user_id, tweaks(JSONB), custom_engines(JSONB) |
| audit_log | 审计日志 | id, ts, actor_id, action, kind, detail(JSONB) |
| app_settings | 应用配置 | key, value(JSONB) |

### （2）用户角色

用户角色存储为 TEXT 类型，约束为：`CHECK (role IN (superadmin, admin, user, guest))`。

### （3）JSONB 字段

NavHub 广泛使用 JSONB 存储灵活配置：

- `widgets.config`：小组件配置（天气城市、音乐源等）
- `user_preferences.tweaks`：用户界面偏好
- `user_preferences.custom_engines`：自定义搜索引擎
- `audit_log.detail`：审计详情

## 5、索引策略

主要索引：

- `groups_owner_idx`：按 owner_id 查询分类
- `groups_pushed_idx`：部分索引，仅索引 pushed=TRUE 的分类
- `icons_group_idx`：(group_id, sort_order) 复合索引
- `widgets_group_idx`：(group_id, sort_order) 复合索引
- `audit_ts_idx`：审计日志按时间倒序
- `audit_kind_idx`：审计日志按类型

## 6、后台维护任务

系统通过后台 Worker 定期执行数据库维护：

- **审计日志清理**：分批删除超过保留期的审计记录（默认 90 天）
- **系统消息清理**：分批删除已过期的系统消息
- **孤立图标清理**：GC 未被引用的临时上传图标及其 S3 对象
